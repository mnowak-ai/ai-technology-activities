Incident #INC-0241 · Government Services Portal
Case 1 — System Running Slowly
Open Opened 9:22 AM 10 pts
What users reported
"Everything is taking forever to load. Pages that normally open in seconds are now taking a full minute or more."
Dashboard updates that used to refresh automatically are now delayed — some users have stopped seeing updates entirely.
Clicking through menus feels sluggish. Buttons respond, but slowly.
No error messages have appeared. The system has not crashed. It's just slow.
Context: Today is the last day of the monthly reporting period. The system normally handles around 40 users at peak times. This morning, over 60 users are logged in at the same time to submit end-of-month reports.
Your investigation
Model answer

Pages that normally load in seconds now take up to a minute, dashboard updates aren't refreshing for some users, and menus feel sluggish. There are no error messages and the system hasn't crashed — everything still works, just slowly.

Model answer

This looks like a capacity issue rather than a bug. It's the last day of the reporting period, and the system has about 60 users logged in instead of its normal 40 — 50% more load than usual. More people doing more work at once likely explains the slowdown.

Model answer

I'd want to know if the slowdown affects every user equally or worse for some, whether it clears up once the reporting deadline passes, and whether this happens every month-end or if today is unusual even for a high-traffic day.

Incident #INC-0242 · Hospital Records Team
Case 2 — Users Cannot Log In Properly
Open Opened 11:47 AM 10 pts
What users reported
Three employees from the hospital records team say they cannot log in on the first try. Their passwords are being rejected, even though they haven't changed them.
One user (account U-118) tried five times before getting in. Another user says their password "just stopped working" this morning.
The system itself is still running. Other users are logged in and working without any problems.
One user eventually got in successfully after multiple attempts. The others have given up and are waiting for help.
Context: Sentinel updated the sign-in process last week. Passwords now need to meet new rules — longer, with a mix of letters and numbers. An announcement was sent by email, but not everyone opened it. Some staff may not know their password no longer meets the new rules.
Your investigation
Model answer

Only three users are affected, and only on their first login attempt — the system itself is working fine for everyone else. One user needed five tries before getting in, and one gave up entirely. This looks isolated to specific accounts, not a system-wide outage.

Model answer

The most likely explanation is the new password rules that went live last week. If these three users didn't see or act on the email announcement, their old passwords may no longer meet the new requirements, causing rejected attempts. It doesn't look like a system bug since other users are logging in normally.

Model answer

I'd want to confirm whether these three users actually read the password-policy email, check if their passwords meet the new rules, and see if a password reset resolves it — that would confirm or rule out the policy change as the cause.

Incident #INC-0243 · School District IT Contact
Case 3 — Strange Behavior, No Clear Cause
Open Opened 2:35 PM 10 pts
What users reported
An error message appears when users try to save files: "Action could not be completed." But if they try again a minute later, it sometimes works fine.
Some users see the error regularly. Others in the same building never see it at all — even doing the same tasks.
The problem started some time this afternoon. Nobody can say exactly when it began or what might have triggered it.
No other errors have been reported alongside this one. The system appears to be working in every other way.
Context: The school district's file storage system was reorganized two days ago — some folders were moved to a new location on the system. Not all user accounts have been updated with access to the new folder location yet. The IT contact is not sure which accounts still need updating.
Your investigation
Model answer

The error is inconsistent — it happens to some users and not others doing the same task, and even for affected users it sometimes works on a retry. That on-and-off pattern usually points to something environmental, like permissions or configuration, rather than a broken feature.

Model answer

The folder reorganization two days ago is the most likely trigger. If some user accounts haven't had their permissions updated to the new folder locations yet, they'd get save errors — while accounts already updated would work fine. That would explain why it's inconsistent between users.

Model answer

I'd want a list of which accounts have and haven't been updated with new folder permissions, and whether the affected users all use the same folder that moved. That would help confirm whether this is a permissions rollout gap rather than something else.
