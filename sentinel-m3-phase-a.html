1Before you write

Three principles of good documentation

Keep these in mind as you work through each incident. Good documentation passes one test: will another analyst understand this without asking you a single question?

Be clear

Use plain language. If a word feels technical, swap it for a simpler one. Pretend you're explaining it to someone on their first day.

"Would someone with no background understand this?"

Be brief

Write only what's needed. Long documentation gets ignored. Short, well-organized documentation gets used. Cut anything that doesn't help someone take action.

"Does every sentence earn its place?"

Be useful

Think about why someone will read this. Future training? Preventing a repeat? Make sure the document serves that purpose directly.

"What will someone do differently after reading this?"

Model documentation — INC-0241
Incident Summary
On the morning of the monthly reporting deadline, the Government Services Portal slowed significantly for all users. Pages that normally load in seconds took a minute or more. The system recovered on its own and no data was lost.
Observed Behavior
Pages loaded very slowly, dashboard updates stopped appearing for some users, and menus responded sluggishly. There were no error messages and no crash — everything worked, just slowly.
Impact
All active users were affected during the slowdown. Because it was the end-of-month deadline, over 60 users were logged in at once — well above the usual 40 — so more people than usual felt the delay.
Resolution
The system returned to normal on its own with no intervention. Heavy user load is the likely cause, but this was never confirmed.
Key Takeaway
Watch user counts ahead of known reporting deadlines. If the slowdown repeats next month under the same conditions, that confirms a capacity pattern worth planning for.

Model documentation — INC-0242
Incident Summary
Three members of the hospital records team could not log in on their first attempts. One account needed five tries before access was granted. The rest of the system worked normally for everyone else.
Observed Behavior
Users saw their passwords rejected even though they hadn't changed them. One user got repeated "invalid password" messages before finally getting in; another gave up and waited for support.
Impact
Three staff were delayed in reaching patient records, and one was fully blocked until helped. Because this is a hospital system, even short access delays matter more than usual.
Resolution
Affected users regained access after resets and retries. The likely cause is the password rule change from the previous week that not all staff knew about — this was never formally confirmed.
Key Takeaway
Before any future password policy change, notify every affected user through more than one channel and give them time to update credentials first. A missed email caused most of today's disruption.

Model documentation — INC-0243
Incident Summary
Some school district users began getting an error when saving files, starting sometime in the afternoon. The error was inconsistent — the same action would sometimes work a minute later, and other users doing the same tasks never saw it at all.
Observed Behavior
The message read "Action could not be completed." Retrying often worked. Who was affected didn't follow an obvious pattern, which made the problem hard for users to describe and hard for support to track.
Impact
Affected users couldn't reliably save their work. Because the error came and went, some people likely never reported it — the true number affected may be higher than recorded.
Resolution
The errors faded on their own later in the day. No fix was applied. The folder reorganization from two days earlier is the suspected cause — some accounts may not have access to the new folder locations — but this was never confirmed or ruled out. This one isn't fully closed.
Key Takeaway
After any reorganization that moves folders or files, review every account's access before calling the work done. Intermittent errors like this are easy to miss until they grow.

Key learnings across all three incidents

Looking at the three incidents together, what patterns or lessons stand out? What do they have in common — and what does that tell the organization?
Model answer

All three incidents trace back to changes or conditions the team knew about in advance — a reporting deadline, a password policy update, a folder reorganization. None were mysteries; they were predictable side effects of planned events that nobody connected ahead of time. The lesson for Sentinel is that known upcoming changes should trigger proactive checks and communication before they cause tickets, not after.

One suggestion for improving future documentation

Based on what you noticed while writing these three summaries — what would make this documentation process easier, clearer, or more useful next time?
Model answer

A simple shared template with the same five fields for every incident would make documentation faster to write and much faster to read — everyone would know exactly where to find the impact or the takeaway. Adding a required "what's still unconfirmed" line would also make honesty about uncertainty the default rather than something each writer has to remember.
